# 툴 사용법 문서 템플릿

핵심 질문: **"이 도구를 어떻게 사용하는가?"**

독자가 문서를 읽고 바로 도구를 사용할 수 있도록, 사전 조건 → 사용 절차 → 트러블슈팅 순서로 구성한다.
다이어그램은 필요한 경우에만 사용한다. 절차가 단순하면 번호 목록만으로 충분하다.

## 출력 구조

````markdown
# {도구/시스템명} 사용법

> 한 줄 요약: {이 도구로 무엇을 할 수 있는지 한 문장으로}

## 사전 조건

| 항목 | 내용 |
|------|------|
| 접근 권한 | {필요한 권한이나 계정} |
| 사전 설치 | {미리 설치해야 하는 것} |
| 설정 | {환경 변수, 설정 파일 등} |

## 사용 절차

### 시나리오 1: {가장 기본적인 사용 시나리오}

1. {첫 번째 단계}: {무엇을 하는지}
2. {두 번째 단계}: {무엇을 하는지}
3. {세 번째 단계}: {무엇을 하는지}

> 결과: {이 절차를 마치면 어떤 상태가 되는지}

### 시나리오 2: {두 번째로 자주 쓰는 시나리오}

1. ...
2. ...

> 결과: ...

## 주요 화면/엔드포인트

| 이름 | 위치 | 용도 |
|------|------|------|
| {화면명 또는 URL} | {접근 경로} | {무엇을 할 수 있는지} |

## 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| {어떤 현상이 발생하는지} | {왜 발생하는지} | {어떻게 해결하는지} |
````

## 정보 요청 예시

기본 질문:
`문서화할 도구명과 핵심 기능을 알려주고, 어떻게 출력하면 되는지와 사용 절차를 파악할 수 있는 문서, 코드, 설정 경로를 보내줘.`

추가 질문:
`설치와 권한 조건, 주요 사용 시나리오, 문제 해결 방법을 확인할 수 있는 경로를 추가로 알려줘.`

## 작성 포인트

- 사용 절차는 **가장 자주 쓰는 시나리오를 먼저** 배치한다.
- 각 단계는 **하나의 동작만** 포함한다. "A를 하고 B를 한다"는 두 단계로 분리한다.
- 절차 사이에 분기가 있으면 별도 시나리오로 분리한다. 한 절차 안에서 "만약 ~이면"을 반복하지 않는다.
- 트러블슈팅은 팀 내에서 실제로 발생한 문제 위주로 적는다. 가상의 문제를 나열하지 않는다.
- 다이어그램은 절차가 복잡하거나 여러 시스템을 오가는 경우에만 추가한다.
- Confluence 위키면 붙여넣기 후 바로 읽히도록 표와 번호 목록 중심으로 정리한다.
