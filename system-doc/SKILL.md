# 시스템 문서화 스킬

팀 내부 시스템을 목적에 맞는 문서로 만든다.
독자는 서버팀 개발자다.
독자가 5분 안에 필요한 정보를 파악하게 만든다.

## 작업 순서

1. 기존 문서 갱신 요청이면 대상 문서를 먼저 확인한다.
2. 문서 유형을 판별한다.
3. 출력 형식을 확인한다.
4. 해당 `references/` 템플릿을 읽는다.
5. 정보가 부족하면 시스템을 파악할 수 있는 경로를 사용자에게 묻는다.
6. 사용자가 제공한 정보와 경로를 기준으로 문서를 작성한다.
7. 갱신 요청이면 변경된 부분만 수정한다.

## 출력 형식

출력 형식은 사용자에게 어떻게 출력할지 먼저 묻고 결정한다.

예를 들어 `Markdown 문서`, `Confluence 위키`, `채팅 본문` 같은 형태를 받을 수 있다.

사용자가 원하는 출력 형식을 지정하면 그 형식을 따른다.

형식을 지정하지 않으면 Markdown으로 작성한다.

다이어그램이 필요하면 Mermaid를 사용한다.

## 문서 유형 판별

사용자의 요청에서 아래 3가지 유형 중 하나를 판별한다. 명확하지 않으면 사용자에게 묻는다.

| 유형 | 핵심 질문 | 키워드 힌트 | 템플릿 |
|------|-----------|-------------|--------|
| **플로우** | "데이터가 어디서 어디로 흘러가는가?" | 흐름, 플로우, 파이프라인, 프로세스, 순서 | `references/flow.md` |
| **아키텍처** | "어떤 컴포넌트가 존재하고 어떻게 연결되는가?" | 아키텍처, 구조, 구성도, 전체 그림, 시스템 구성 | `references/architecture.md` |
| **툴 사용법** | "이 도구를 어떻게 사용하는가?" | 사용법, 사용 방법, 가이드, 설정 방법, 어떻게 쓰는지 | `references/tool-usage.md` |

유형을 판별한 뒤 해당 `references/` 파일을 읽고 그 템플릿을 따른다.

## 정보 수집

정보가 부족하면 시스템을 파악할 수 있는 문서, 코드, 설정 경로를 질문한다.

한 번에 많은 질문을 하지 않는다.
문서 작성에 필요한 핵심 경로만 묻는다.

질문은 세부 사실보다 파악 경로를 묻는 형태로 작성한다.

### 질문 방식

유형별 질문 예시는 해당 `references/` 템플릿의 "정보 요청 예시" 섹션을 따른다.

## 공통 작성 원칙

유형에 관계없이 모든 문서에 적용한다.

### 문장

- **"A가 B에게 C를 요청한다"** 형태의 주어-동사 구조로 쓴다.
- 종결 어미는 **"~한다"**로 통일한다.
- 사실만 서술한다. 권유형("~하는 것이 좋다"), 추측형("~일 수 있다")을 쓰지 않는다.
- 사용자가 직접 제공한 정보와 사용자가 안내한 경로에서 확인한 정보만 단정한다.
- 관계가 명시되지 않으면 연결을 추정해서 쓰지 않는다.
- 정보가 부족하면 문서 작성을 멈추고 필요한 경로를 다시 묻는다.

### 용어 통일

- 같은 컴포넌트는 문서 전체에서 동일한 이름으로 지칭한다.
- 다이어그램 노드명, 본문, 테이블의 컴포넌트명이 모두 일치해야 한다.
- 사용자가 제공한 팀 내부 명칭이 있으면 그 이름을 우선 사용한다.
- 팀 내부 명칭이 없으면 사용자가 제공한 식별자를 그대로 사용한다.

### 다이어그램

- Mermaid(`sequenceDiagram`, `flowchart`, `graph`)를 사용한다.
- 노드명은 **팀에서 부르는 한글 서비스명**을 그대로 쓴다.
- 화살표 라벨에 이동하는 데이터를 간결히 적는다.
- 한 다이어그램당 노드 **10개 이하**. 넘으면 분리한다.
- 다이어그램에 표현한 내용을 텍스트에서 반복하지 않는다.
- 다이어그램 시각화가 필요하면 `mmdc`(mermaid-cli)를 사용한다. `mmdc`가 설치되지 않은 환경이면 Mermaid 코드 블록만 제공한다.

### 분량

- 전체 문서를 **5분 이내**에 읽을 수 있는 분량을 유지한다.
- 한 섹션이 화면 하나(약 40줄)를 넘기면 분리하거나 줄인다.

### 금지 사항

| 금지 항목 | 이유 |
|-----------|------|
| 기술 스택 선정 이유, 도입 배경 | 문서 목적과 무관 |
| 내부 구현 상세 (알고리즘, 자료구조, 클래스) | 문서 목적과 무관 |
| 코드 블록 | 함수명이나 엔드포인트는 인라인 코드로만 언급 |
| "개요", "소개" 같은 빈 도입부 | 본론 진입을 지연시킴 |
| 장단점 비교, 대안 분석 | 별도 문서의 영역 |

### 정보 계층

독자가 위에서 아래로 읽으며 이해가 점진적으로 깊어지도록 구성한다.

```
다이어그램  →  전체 그림을 30초 안에 파악
상세 설명  →  각 항목의 세부사항 확인
테이블      →  빠른 조회용 정보
```

## 완성 전 체크리스트

문서를 저장하기 전에 확인한다. 하나라도 실패하면 수정 후 저장한다.

- [ ] 다이어그램만 보고 전체 구조/흐름을 파악할 수 있는가?
- [ ] 다이어그램, 본문, 테이블의 컴포넌트명이 모두 일치하는가?
- [ ] 기술 원리나 구현 상세를 설명하는 문장이 없는가?
- [ ] 모든 문장이 "~한다" 종결 어미를 사용하는가?
- [ ] 5분 이내에 읽을 수 있는 분량인가?
- [ ] 해당 유형의 템플릿 구조를 빠짐없이 따랐는가?

## 저장

- 파일명: `{시스템명-영문}.md`
- Markdown 문서면 사용자가 저장 경로를 지정하지 않을 때 현재 작업 디렉토리에 저장한다.
- Confluence 위키면 저장 경로 대신 게시 위치나 위키 공간을 사용자에게 확인한다.
- 채팅 본문이면 별도 파일 저장 없이 응답 본문으로 제공한다.
- 기존 문서 갱신 요청인 경우 대상 문서를 사용자에게 확인한 뒤 변경된 부분만 수정한다.
