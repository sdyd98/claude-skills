# 사내 Claude Code 하네스 비교 분석 지시서

## 역할

너는 사내 Claude Code 하네스와 공식 레퍼런스 브리핑을 **객관적으로** 비교 분석하는 에이전트다. 목표는 "좋다/나쁘다" 판정이 아니라 **"커버/미커버" 식별 + 팀 제약 기반 우선순위 Gap 리스트** 산출.

---

## 입력 3개

1. **사내 하네스** — 현재 저장소의 구성:
   - `./CLAUDE.md`, `./CLAUDE.local.md` (있으면)
   - `./.claude/` 디렉토리 전체 (`settings.json`, `rules/`, `skills/`, `commands/`, `agents/`, 기타)
   - 루트·하위 디렉토리의 `CLAUDE.md` 모두
   - 관련 설정·스크립트 (`.github/`, CI 설정 중 Claude Code 관련)

2. **레퍼런스 브리핑** — `_review/harness-design-brief.md`:
   - §1 팀 맥락·제약
   - §2 7축 설계 프레임
   - §3 공식 확장점 카탈로그 (2026-04 기준)
   - §4 Phase 전략 원칙
   - §5 검증·측정 프레임

3. **팀 제약** (브리핑 §1.3 요약):
   - C++/MSVC/VS, 580만 줄, .lib 모듈 → GameServer.exe
   - **테스트 인프라 현재 0**
   - 사내 codegen 3종 (패킷/DB/xls)
   - 6인 팀, GoogleTest/Catch2 경험 0명
   - 보안 정책 승인됨
   - Claude Code + Codex 서브

## 수행 절차

### Step 1 — 인벤토리 수집 (Read tool 사용)
- 사내 하네스 파일을 **실제로 읽어** 모든 구성 요소 목록화
- 추측 금지: 읽은 것만 기록
- 읽기 실패·접근 불가는 **"확인 불가"**로 명시

### Step 2 — 7축 매트릭스 매핑
- 브리핑 §2의 7축 각각에 대해 사내 하네스의 커버리지 평가
- 4단계 척도: **없음 / 기초 / 중간 / 완성**
- 각 평가는 **근거 파일·경로**와 함께

### Step 3 — 공식 카탈로그 대조
- 브리핑 §3의 각 확장점(3.1~3.11)에 대해 사내 사용 여부 판정
- 사용 중이면 어떤 수준인지
- 미사용이면 **"의도적 제외"**인지 **"인지 못 함"**인지 단서가 있는가

### Step 4 — Gap 리스트 작성
- 팀 제약(테스트 0, 580만 줄 등)을 고려한 우선순위
- P0(즉시) / P1(Phase 1) / P2(Phase 2) / P3(장기)
- 각 Gap에 대해 팀 제약과의 관련성 명시

### Step 5 — Keep / Drop / Add 정리
- **Keep**: 현재 있고 기능이 유효 (근거 있을 때만)
- **Drop**: 공식이 더 나은 대안을 제공하나 사내는 구식
- **Add**: 공식에 있고 팀 제약상 필요하나 사내에 없음

### Step 6 — 팀 확인 질문 수집
- 설계 의도·히스토리상 파일만으로 판단 못하는 항목
- 5~10개

## 제약 사항 (반드시 준수)

1. **실제로 읽은 파일만 근거로 사용.** Read tool 호출 결과만 인용.
2. 읽을 수 없는 부분은 **"확인 불가"**로 명시하고 추측하지 말 것.
3. **"좋다/나쁘다" 판단 금지.** "커버/미커버", "있음/없음", "추정/단정"만 사용.
4. 사내 하네스의 설계 결정은 **팀 맥락·과거 경험**이 반영됐을 수 있음을 항상 고려.
5. 브리핑 §3 카탈로그는 **공식 기능 목록**이지 **필수 구현 목록이 아님**. 미사용이라고 해서 누락이라 단정하지 말 것.
6. 버전 차이로 공식 기능이 사내에 미지원일 수 있음 — 확인 불가 시 "버전 확인 필요"로 표기.
7. 추론·추정은 **"추정: …"** 접두어 명시.
8. 출력 전 자기 검토: 6개 제약을 모두 준수했는가?

## 결과 활용 주의

이 분석 결과는 **토론 자료**다. 자동 실행·자동 개선 대상이 아님.
- 결과를 팀 리뷰 미팅에 올려 **대면 논의**로 결정
- 사내 하네스 변경은 팀 합의 후에만
- P0/P1 Gap은 팀장 보고 후 다음 스프린트 일감으로

## 실행 방법 예시

### 대화형 실행
```
> _review/compare-prompt.md 에 따라 비교 분석을 수행해.
> 브리핑: _review/harness-design-brief.md
> 결과는 _review/comparison-result.md 로 저장.
```

### Non-interactive 실행 (CI·배치)
```bash
claude -p "$(cat _review/compare-prompt.md)
브리핑: $(cat _review/harness-design-brief.md)" \
  --output-format markdown \
  > _review/comparison-result.md
```

### 결과 파일 위치
- `_review/comparison-result.md` (gitignored 권장)
- 팀 공유는 결과 파일을 정제한 후 별도 문서로
