# 사내 Claude Code 하네스 3자 비교 분석 지시서

## 역할

너는 **3자 비교 분석**을 수행하는 에이전트다. 목표는 "좋다/나쁘다" 판정이 아니라 **"커버/미커버" 식별 + 팀 제약 기반 우선순위 Gap 리스트** 산출.

---

## 입력 3개

### 1. 사내 하네스 (현재 상태)
현재 저장소의 구성:
- `./CLAUDE.md`, `./CLAUDE.local.md` (있으면)
- `./.claude/` 디렉토리 전체 (`settings.json`, `rules/`, `skills/`, `commands/`, `agents/`, 기타)
- 루트·하위 디렉토리의 `CLAUDE.md` 모두
- 관련 설정·스크립트 (`.github/`, CI 설정 중 Claude Code 관련)

### 2. 레퍼런스 브리핑 (중립 기준)
`_review/harness-design-brief.md`:
- §1 팀 맥락·제약
- §2 7축 설계 프레임
- §3 **공식 확장점 카탈로그 (2026-04 기준)** ← 중립 사실
- §4 Phase 전략 원칙
- §5 검증·측정 프레임
- §6 비교 주의사항

### 3. 팀 내부 제안 (특정 제안)
`_review/our-proposal.md`:
- 2026-04 내부 논의 기반 설계안
- 4계층 구조, Track A/B 매핑, Phase 1 MVP, `/write-tests` 우선 등
- **정답이 아닌 논의 산출물**

## 3자의 성격 차이 (중요)

| 입력 | 성격 | 권위 |
|--|--|--|
| 사내 하네스 | **현실** — 실제 운영 중 | — |
| 레퍼런스 브리핑 | **중립 사실** — 공식 기능 카탈로그 | 높음 (팩트) |
| 팀 내부 제안 | **특정 제안** — 논의 산출물 | 낮음 (의견) |

**절대 원칙**: 레퍼런스 브리핑과 팀 내부 제안을 **동등한 기준**으로 취급하지 말 것. 브리핑은 팩트, 제안은 의견.

## 팀 제약 (비교 우선순위 기준)

- C++/MSVC/VS 게임서버, 580만 줄
- `.lib` 모듈 → `GameServer.exe` 단일
- **테스트 인프라 현재 0**
- 사내 codegen 3종 (패킷·DB·xls)
- 6인 팀, GoogleTest/Catch2 경험 0명
- 보안 정책 승인됨
- Claude Code + Codex 서브

## 수행 절차

### Step 1 — 인벤토리 수집 (Read tool)
사내 하네스 파일을 **실제로 읽어** 모든 구성 요소 목록화. 추측 금지.

### Step 2 — 7축 매트릭스 매핑
브리핑 §2의 7축 × 4단계 척도 (없음/기초/중간/완성). 근거 파일 필수.

### Step 3 — 공식 카탈로그 대조 (브리핑 §3)
각 확장점(3.1~3.11)에 대해 사내 사용 여부. **이건 절대 기준**.

### Step 4 — 팀 내부 제안과의 대조 (`our-proposal.md`)
팀 제안의 각 요소에 대해 사내 사용·미사용 식별. **동의/불동의 판정 금지**. 단순히 **차이**만 기술.

### Step 5 — Gap 리스트
팀 제약 반영 우선순위 (P0~P3).
- **공식 카탈로그 미커버**: 구조적 Gap
- **팀 제안과 다름**: 차이일 뿐, 반드시 Gap 아님

### Step 6 — Keep / Drop / Add
- **Keep**: 현재 있고 유효
- **Drop**: 공식에 더 나은 대안 존재 (단, 의도적 선택 가능성 열어둘 것)
- **Add**: 공식에 있고 팀 제약상 필요하나 사내 없음

### Step 7 — 팀 제안에 대한 평가 (중립)
팀 제안 자체를 비평. 사내 하네스가 오히려 더 나은 점이 있다면 **명시**.

### Step 8 — 팀 확인 질문
파일만으로 판단 불가한 항목 5~10개.

## 제약 사항 (반드시 준수)

1. **실제로 읽은 파일만** 근거. Read tool 호출 결과만 인용.
2. 읽을 수 없는 부분은 **"확인 불가"** 명시, 추측 금지.
3. **"좋다/나쁘다" 판단 금지.** "커버/미커버", "있음/없음", "추정/단정"만.
4. 사내 하네스의 설계 결정은 **팀 맥락·과거 경험** 반영 가능성 항상 고려.
5. 브리핑 §3 카탈로그는 **공식 기능 목록**이지 **필수 구현 목록 아님**.
6. **팀 제안(`our-proposal.md`)은 의견**. 이를 절대 기준으로 취급 금지.
7. 버전 차이로 공식 기능이 사내에 미지원일 수 있음 — "버전 확인 필요" 표기.
8. 추론·추정은 **"추정: …"** 접두어 명시.
9. 팀 제안의 약점도 **의도적으로 찾아** 섹션 7에 명시 (균형 유지).
10. 출력 전 자기 검토: 9개 제약 모두 준수했는가?

## 결과 활용 주의

이 분석 결과는 **토론 자료**다. 자동 실행·자동 개선 대상 아님.
- 결과를 팀 리뷰 미팅에 올려 **대면 논의**
- 사내 하네스 변경은 팀 합의 후에만
- P0/P1 Gap은 팀장 보고 후 다음 스프린트 일감으로
- 팀 제안도 팀 리뷰에서 수정 가능

## 실행 방법 예시

### 대화형
```
> _review/compare-prompt.md 에 따라 3자 비교 분석을 수행해.
> 레퍼런스: _review/harness-design-brief.md
> 팀 제안: _review/our-proposal.md
> 결과: _review/comparison-result.md 로 저장.
```

### Non-interactive (CI·배치)
```bash
claude -p "$(cat _review/compare-prompt.md)
레퍼런스: $(cat _review/harness-design-brief.md)
팀 제안: $(cat _review/our-proposal.md)" \
  --output-format markdown \
  > _review/comparison-result.md
```

### 결과 파일 위치
- `_review/comparison-result.md` (gitignored 권장)
- 팀 공유는 결과 파일을 정제한 후 별도 문서로
